# AGENTS.md

## 1. このファイルの目的

このファイルは、現実派AI副業診断サイトのリポジトリで作業するCodexおよびAIエージェント向けの恒久ルールです。

このリポジトリ内で調査・計画・実装・テスト・画面確認を行う場合は、最初にこのファイルを読み、ここに書かれた方針を優先してください。

個別の依頼に、このファイルと異なる明示的な指示がある場合は、勝手に判断せず、矛盾点を報告して確認を待ってください。

## 2. サイトの目的

このサイトは、AI副業に興味はあるものの、何から始めればよいか分からない初心者に向けた無料の診断サイトです。

診断を通して、利用者が次の内容を理解し、小さな行動へ進める状態を目指します。

- 自分に合いやすいAI副業の方向性
- 向いている稼ぎ方と向いていない稼ぎ方
- 今日できる小さな一歩
- 7日間の行動計画
- 30日後に目指す完成物
- 最初の商品案
- お金につなげる現実的な順番
- 最初に使うAIツール
- 課金するかどうかの判断基準

このサイトは、成功者が上から教える場所ではありません。

運営者自身も診断結果を使ってAI副業を試し、成功だけでなく、失敗・迷い・改善も記録する「実験ログ型」のサイトです。

最初から大きく稼ぐことではなく、次の流れを繰り返すことを重視します。

1. 小さく作る
2. 公開する
3. 反応を見る
4. 改善する
5. 記録する

## 3. 対象ユーザー

主な対象ユーザーは、過去の運営者に近い次のような人です。

- 副収入が欲しい
- AIには興味がある
- 副業の知識は少ない
- 何から始めればよいか分からない
- 行動の最初の一歩が重い
- 飽きやすく、継続が苦手
- 抽象的な説明では動けない
- 専門用語を見ると離脱しやすい
- 少し分からないことがあると止まりやすい
- 大きな目標より、小さな達成感が必要
- 怪しいAI副業や過度な収益アピールに不安がある

実装や文章を判断する際は、知識のある制作者ではなく、上記の初心者がスマートフォンで初めて使う場面を基準にしてください。

## 4. 基本の役割分担

このプロジェクトでは、GPTとCodexの役割を次のように分けます。

### GPT

- プロダクトの目的を決める
- 優先順位を決める
- ユーザー視点で判断する
- 変更禁止事項を決める
- 合格条件を決める
- Codexの結果を評価する
- 最終的な公開判断を行う

### Codex

- リポジトリを調査する
- 必要に応じて実装計画を作る
- コードを変更する
- テストする
- ブラウザとスマホ幅で確認する
- 差分を自己レビューする
- 定型形式で完了報告を行う

Codexは、コードを書いただけで作業を完了したことにしてはいけません。

## 5. GPT-5.6対応の作業ルール

### 5.1 タスクの大きさに応じて進め方を変える

#### 小さな修正

例：

- 文言変更
- リンク差し替え
- 押せないボタンの削除
- 小さな余白調整
- 1ファイル内の軽微な修正

進め方：

1. 関連箇所を調査
2. 最小限の実装
3. テスト
4. 画面確認
5. 完了報告

Planモードは原則不要です。

#### 通常の改修

例：

- 複数ファイルにまたがるUI改善
- 結果ページの表示整理
- コピー・シェア機能の改善
- 複数タイプ共通の表示変更

進め方：

1. 関連実装を調査
2. 短い実装方針を整理
3. 必要な範囲だけ変更
4. テスト
5. 画面確認
6. 自己レビュー
7. 完了報告

#### 複雑な改修

例：

- 診断結果ページ全体の再構成
- 大量のレビュー項目への対応
- 状態保存や復元機能
- 多数のファイルへ影響する変更

必ずPlanモードから始めてください。

進め方：

1. コードを変更せず調査
2. 依存関係を整理
3. 最大3段階の計画を作成
4. GPTまたは人の承認を待つ
5. 承認された第1段階だけ実装
6. テストと画面確認
7. GPTへ結果を戻す

大量のレビュー項目を一括で修正してはいけません。

基本方針は、**計画は大きく、実装は小さく**です。

### 5.2 1回の依頼を広げすぎない

原則として、1回の依頼は1つの目的に限定してください。

次の作業を一度に詰め込まないでください。

- 調査
- 大規模実装
- 長文校正
- ブラウザ操作
- テスト
- 公開確認
- push
- デプロイ

作業量が大きい場合は、調査・実装・確認に分けてください。

## 6. 初心者向けの文章ルール

サイト内の文章は、初心者が一度読んで意味を理解できることを優先します。

### 必須ルール

- 中学生でも理解しやすい日本語を使う
- 1文を長くしすぎない
- 抽象的な説明だけで終わらせない
- 「誰が」「何を」「どこで」「どうするか」を具体的にする
- 今日やることが分かる文章にする
- 大きな成功より、小さな完成物を示す
- 成功者の上から目線にしない
- 運営者も試行錯誤中である姿勢を維持する
- 強い断定や煽りを避ける
- 同じ意味の説明を何度も繰り返さない
- スマホで流し読みしても要点が分かるよう、見出しと箇条書きを使う

### 避ける表現

次のような表現は、明示的な指示がない限り使用しないでください。

- 誰でも稼げる
- 必ず稼げる
- 絶対に成功する
- 月100万円
- 放置で稼げる
- 今だけ
- 残り〇名
- 簡単に人生が変わる
- この方法だけで十分
- 失敗しない

## 7. 専門用語の扱い

専門用語は、原則として一般的な日本語へ言い換えてください。

### 推奨する言い換え

- MVP → 最初の試作品 / まず動く形
- CTA → 次にやること / 次の行動
- マネタイズ → お金につなげる流れ
- LP → 商品説明ページ
- テンプレート / テンプレ → ひな形
- ロードマップ → 行動計画
- コンバージョン → 申込みや購入
- リード → 興味を持ってくれた人
- リファクタリング → 動きを変えずにコードを整理すること

専門用語を残す必要がある場合は、最初に出る場所で短い補足を付けてください。

例：

> MVP（まず動く最初の試作品）

画面上に専門用語を増やす変更は、明示的な承認なしで行わないでください。

## 8. 収益に関する方針

このサイトは収益を保証しません。

実装・文章・診断結果では、次の方針を守ってください。

- 特定の収入額を保証しない
- 再現性を断定しない
- 誰にでも同じ結果が出るように見せない
- AIツールを使えば自動で稼げると説明しない
- 最初の目的は「作って公開する成功体験」とする
- 無料配布や100〜300円程度の小さな商品から試す流れを基本とする
- 反応を確認してから次の段階を検討する
- AIツールやサービスを紹介する場合は、必要に応じてPR表記を行う
- 投資や高レバレッジ取引をAI副業の収益手段として推奨しない

診断結果は、適性や成功を確定するものではなく、最初の方向性を整理するための参考情報として扱ってください。

## 9. UI・UXの基本方針

### 9.1 スマホ優先

このサイトはスマートフォンでの利用を最優先にしてください。

デスクトップで見栄えがよくても、スマートフォンで読みづらい変更は不合格です。

### 9.2 必須ルール

- 横スクロールを発生させない
- 文字を小さくしすぎない
- ボタンは親指で押しやすい大きさにする
- ボタン同士を近づけすぎない
- 固定ボタンが本文や選択肢へ重ならないようにする
- 長い文章を1つのカードへ詰め込まない
- 見出しだけでも内容の順番が分かるようにする
- 重要な結論と「次にやること」を先に見せる
- 情報を増やすだけでなく、重複を減らす
- 押せない要素をボタンのように見せない
- 外部リンクであることが分かる文言にする
- 結果ページでは、ユーザーが次に進める実在する導線を用意する
- 色だけに頼って状態を伝えない
- フォーカス表示を消さない
- 画像やアイコンには必要に応じて代替テキストを設定する

### 9.3 基準幅

最低限、次の幅で確認してください。

- 375px
- 430px
- デスクトップ幅

## 12. 回答データとプライバシー

診断の回答内容は、原則としてブラウザ内だけで処理してください。

### 禁止事項

明示的な承認なしに、次を行ってはいけません。

- 回答データを外部サーバーへ送信する
- 回答データを外部APIへ送信する
- 回答データを分析サービスへ送信する
- 回答データをURLへ含める
- 回答データを問い合わせフォームへ送る
- 個人情報と回答を結び付ける
- メールアドレス、氏名、電話番号、住所を収集する
- 回答を永続保存する仕組みを追加する
- localStorage、sessionStorage、Cookieへ回答を保存する

ブラウザ内の保存が必要な場合も、個別の承認を得てから実装してください。

### 説明文のルール

「回答は外部送信されません」などの説明を画面へ追加する場合は、必ず実装を確認し、事実と一致する表現だけを使用してください。

外部アクセス解析や外部スクリプトが存在する場合は、回答データの送信とページ閲覧情報の送信を混同しないでください。

明示的な承認なしに、外部APIや新しい外部サービスを追加してはいけません。

対象例：

- 生成AI API
- 分析API
- メール配信サービス
- フォームサービス

- 認証サービス

- 広告・トラッキングサービス

- 推測でURLを作らない
- 新しいタブで開く場合は `target="_blank"` を使用する
- `rel="noopener noreferrer"` を付ける
- リンク切れを確認する
- アフィリエイトリンクの場合は必要なPR表記を確認する
- 外部サービスへの送信内容を確認する

## 14. セキュリティ

- APIキー、トークン、パスワード、個人情報をコードへ記載しない
- 秘密情報をGitへ追加しない
- `.env` を公開対象へ含めない
- HTMLへ外部入力を挿入する場合は、XSSの危険を確認する
- `innerHTML` の新規使用は必要性を説明し、安全性を確認する
- 外部URLは適切にエンコードする
- コピー・シェア文は `URLSearchParams` または適切なエンコード処理を使用する
- 不要な権限を求めない
- セキュリティ上の問題を見つけた場合は、範囲外でも報告する。ただし、勝手に大規模修正しない

## 16. 変更範囲の原則

- 目的に必要な最小範囲だけ変更する
- 関係のないリファクタリングを行わない
- ファイル名やディレクトリ構成を勝手に変更しない
- 既存の文体やデザインを無断で全面変更しない
- 複数の改善を一度に詰め込まない
- 「ついでの改善」を行わない
- 既存コードを再利用できる場合は再利用する
- 新しい抽象化は、今回の目的に必要な場合だけ追加する
- 変更理由を説明できない差分を残さない

## 17. テストルール

リポジトリに既存のテスト・ビルド・Lint・構文チェックがある場合は、関連するものをすべて実行してください。

コマンドは、`package.json`、README、既存スクリプト、CI設定を確認し、リポジトリで定義されているものを使用してください。

存在しないコマンドを推測で実行しないでください。

### 17.1 必須確認項目

変更内容に応じて、最低限次を確認してください。

#### 診断フロー

- トップページが表示される
- 診断を開始できる
- すべての質問へ回答できる
- 「次へ」が動く
- 「戻る」が動く
- 進捗表示が正しい
- 未回答時の挙動が正しい
- 最終回答後に結果が表示される
- もう一度診断できる

#### 診断ロジック

- 変更前後で同じ回答が同じタイプになる
- すべての既存タイプを表示できる
- スコア表示が正しい
- 満点表記がある場合は実際の最大値と一致する
- 同点処理が変わっていない
- `NaN`、`undefined`、空表示がない

#### 結果ページ

- タイプ名が表示される
- 結果文が表示される
- 今日やることが分かる
- 7日間の行動計画が読める
- 30日後の目標が読める
- リンクが正しい
- 押せないボタンが残っていない
- 結果コピーが動く
- コピー成功・失敗表示が分かる
- Xシェアが動く
- シェア文の日本語と改行が崩れない
- 外部リンクが安全に開く

#### プライバシーと通信

- 回答時に外部送信が発生しない
- 不要なネットワークリクエストがない
- 新しい外部APIが追加されていない
- コンソールに秘密情報が出ない
- URLへ回答内容が露出しない

#### 表示

- 横スクロールがない
- 文字切れがない
- ボタンの重なりがない
- 固定要素が本文へかぶらない
- 長い見出しが崩れない
- カードや表が画面幅を超えない
- 外部リンク文言が分かる
- ダークテーマの可読性が保たれている

## 20. アクセシビリティの最低基準

- ボタンには内容が分かる文言を使用する
- アイコンだけのボタンにはラベルを付ける
- キーボードで操作できる状態を維持する
- フォーカス表示を消さない
- 見出し階層を飛ばさない
- `button` と `a` を用途に応じて正しく使う
- フォーム要素にはラベルを関連付ける
- 色だけで選択状態やエラーを示さない
- 十分な文字と背景のコントラストを維持する
- 画像へ必要な代替テキストを付ける

## 21. Gitルール

### 21.1 作業前

- 現在のブランチを確認する
- `git status` を確認する
- 既存の未コミット変更を確認する
- ユーザーの変更を勝手に破棄しない
- 既存の差分が今回の作業と無関係な場合は触らない

### 21.2 作業中

- 関係のないファイルを変更しない
- 強制操作を行わない
- `git reset --hard` を実行しない
- ユーザーの変更を上書きしない
- 秘密情報をステージしない
- 生成物やキャッシュを不要にコミットしない

### 21.3 commit

個別指示で許可されている場合だけcommitしてください。

commit前に次を確認してください。

- 差分が今回の目的だけに限定されている
- テストが成功している
- 画面確認が完了している
- 秘密情報が含まれていない
- 不要なファイルが含まれていない

commitメッセージは、変更内容が分かる短い英語または既存リポジトリの方針に合わせてください。

### 21.4 push・デプロイ

明示的な指示がない限り、次を行ってはいけません。

- `git push`
- force push
- GitHub Web editorでの公開反映
- GitHub Pagesの設定変更
- デプロイ
- 公開URLの更新
- タグ作成
- リリース作成

最終的な公開判断はGPTまたは人が行います。

「実装完了」と「公開完了」を混同しないでください。

---

## 22. コードを書いただけでは完了ではない

次の状態だけでは、作業完了としてはいけません。

- コードを書いた
- ファイルを保存した
- 構文エラーがない
- ビルドが通った
- 1つの画面だけ確認した
- デスクトップだけ確認した

最低限、次が必要です。

1. 目的を満たす実装
2. 変更範囲の自己レビュー
3. 既存機能を壊していない確認
4. 利用可能なテストの実行
5. 375pxでの画面確認
6. 430pxでの画面確認
7. デスクトップ確認
8. コンソール確認
9. 必要に応じたネットワーク確認
10. 外部リンク確認
11. 未確認事項の報告
12. 定型形式の完了報告

確認できなかったことは、正直に「未確認」と報告してください。

## 23. 停止条件

次の状況では、推測で作業を続けず停止してください。

- 個別指示とこのファイルが矛盾している
- 診断ロジック変更が必要になる
- 質問数や配点への影響が避けられない
- 公開URLが確認できない
- 利用するnoteやXのURLが特定できない
- 外部送信の有無を確認できない
- 秘密情報が必要
- 既存の未コミット変更と競合する
- 元へ戻せない画面操作が必要
- GitHub Pagesの構成を壊す可能性がある
- テスト失敗の原因が今回の変更か判別できない
- 明示されていないpushやデプロイが必要
- 作業範囲が依頼より大きくなる

停止時は、次を報告してください。

- どこまで完了したか
- 何が確認できなかったか
- 続行に必要な判断
- 安全な次の選択肢
