# 開発運用ガイド

このガイドは、「現実派AI副業診断」を今後Codexだけで安全に継続開発するためのルールです。
大きく作り替えるより、診断体験を少しずつ安定して改善することを優先します。

## プロジェクト概要

「現実派AI副業診断」は、AI副業初心者向けの無料診断サイトです。
ユーザーが10問の質問に答えると、4つのAI副業タイプのうち最も合うタイプを表示し、月1万円を現実的に目指すための最初の行動を提示します。

診断タイプは以下の4つです。

- AIライティング・コンテンツ型
- AI画像/POD・クリエイター型
- ノーコード/ミニツール開発型
- リサーチ・まとめメディア型

## 技術スタック

- Vite
- React
- TypeScript
- Vitest
- フロントエンドのみ
- API接続なし
- スマホ優先UI

## 編集場所

質問文を編集する場所:

- `src/data/questions.ts`
- 質問文、補足文、選択肢、選択肢ごとのスコアを編集します。
- 質問データをコンポーネント内に直書きしないでください。

診断結果文を編集する場所:

- `src/data/results.ts`
- タイプ名、一言説明、強み、弱み、向いているAI副業、向いていないAI副業、最初の3ステップ、7日間ロードマップ、まず使うAIツールを編集します。
- 結果文を `ResultPage.tsx` に直書きしないでください。

診断ロジックを編集する場所:

- `src/utils/calculateResult.ts`
- スコア集計、未回答時の扱い、同点時の優先順位を管理します。
- 同点時の優先順位は `resultTypePriority` です。
- 現在の優先順は `tool` → `research` → `writing` → `creative` です。
- 理由なく診断ロジックを大きく変更しないでください。

CTAを編集する場所:

- `src/data/cta.ts`
- CTA見出し、説明文、ボタン文言、準備中メッセージ、将来の遷移先URLを編集します。
- アフィリエイトリンクや外部URLをコンポーネント内にハードコードしないでください。

30日ロードマップを編集する場所:

- `src/data/roadmaps.ts`
- 30日後の成果物、1週目から4週目のテーマ、各週の行動を編集します。
- CTAの実URLを入れる前に、まずこの中身を具体化します。

Xシェア文を編集する場所:

- `src/data/share.ts`
- シェア本文、ハッシュタグ、公開後に入れるサイトURLを編集します。
- 公開URLが決まるまでは `siteUrl` 未設定のままで問題ありません。

## 新機能追加時の注意点

- まず既存MVPを壊さないことを優先します。
- 変更は小さく分け、質問、結果、CTA、表示コンポーネントの責務を混ぜないでください。
- 新しい診断タイプを追加する場合は、`ResultType`、質問スコア、結果データ、CTAデータ、テストをセットで更新します。
- 外部サービス連携、ルーティング追加、フォーム送信、API連携は、明示的に依頼された場合だけ実装します。
- スマホ表示で読みにくい長文にならないように、1項目は短めに保ちます。
- 「誰でも簡単に稼げる」など、煽りや誤解を招く表現は避けます。

## 毎回必ず実行するコマンド

Windows PowerShellでは `npm` ではなく `npm.cmd` を使ってください。

```bash
npm.cmd run build
npm.cmd test
```

依存関係を入れる場合もPowerShellでは以下を使います。

```bash
npm.cmd install
```

## build/test確認手順

1. 変更したファイルを確認する。
2. `npm.cmd run build` を実行する。
3. `npm.cmd test` を実行する。
4. 失敗した場合は原因を直し、もう一度 build/test を実行する。
5. 完了報告では build/test の結果を必ず書く。

build/testは並列実行せず、この順番で実行してください。
`dist/` と `*.tsbuildinfo` は生成物なので、コミット対象にしないでください。

## 公開前チェック

公開前は [PUBLISH_CHECKLIST.md](PUBLISH_CHECKLIST.md) を確認します。
公開方法を決める場合は [DEPLOYMENT_GUIDE.md](DEPLOYMENT_GUIDE.md) を確認します。
本番URL、課金、アフィリエイト、メール登録導線を入れる場合は、ユーザー確認が必要な変更として扱ってください。
公開URLやCTA本番URLの設定手順は [URL_SETUP_GUIDE.md](URL_SETUP_GUIDE.md) を確認します。

## Git管理を始める前のチェック

Git管理を始める場合は [GIT_START_CHECKLIST.md](GIT_START_CHECKLIST.md) を確認します。
Codexはユーザーから明示的に依頼されるまで、`git init` やコミット作成は行いません。
リモートリポジトリへ接続する場合は [REMOTE_REPO_GUIDE.md](REMOTE_REPO_GUIDE.md) を確認します。

## Codexが自動で進めてよい条件

以下をすべて満たす作業は、ユーザー確認なしで次に進めます。

- 文言修正
- READMEやdocsの整理
- テスト追加
- 型定義の整理
- 小さなUI改善
- build/testが成功している
- 既存機能を壊していない

## ユーザー確認が必要な条件

以下に当てはまる場合は、作業を進める前または完了時に確認必要として報告します。

- 診断タイプを増減する
- 診断ロジックを大きく変更する
- 外部サービス連携を追加する
- 課金/アフィリエイト/メール登録導線を本番URLにする
- デザインを大きく変える
- ファイル構成を大きく変える
- build/testが失敗している

## 完了報告フォーマット

完了報告は長文にせず、以下の形式に統一します。

```text
【判定】
OK / 要確認 / NG

【一言まとめ】
何をしたかを1〜2行で説明

【あなたの確認が必要か】
確認不要 / 確認必要

【確認が必要な場合】
見るべき箇所を最大3つ

【実行結果】
* build：成功 / 失敗
* test：成功 / 失敗

【変更ファイル】
最大5個まで。多い場合は重要なものだけ

【次にやるべきこと】
優先順位1位だけ
```

## Codexへ依頼するときのテンプレート

```text
目的：

現状：

今回やってほしいこと：

やってはいけないこと：

完了条件：

報告してほしいこと：
```

依頼例:

```text
目的：
診断結果ページを、初心者が次の行動を決めやすい内容にしたい。

現状：
4タイプの結果ページはあるが、CTAは仮実装のまま。

今回やってほしいこと：
src/data/cta.ts の文言を改善し、ResultPage.tsx の表示崩れがないか確認してください。

やってはいけないこと：
外部サービス連携やルーティング追加はしないでください。

完了条件：
npm.cmd run build と npm.cmd test が通ること。

報告してほしいこと：
修正したファイル、build結果、test結果、残っている課題。
```

## やってはいけないこと

- 大きな機能追加はしない
- 既存の診断ロジックを理由なく大きく変えない
- 質問データと結果データをコンポーネント内に直書きしない
- build/testが通らない状態で完了しない
- 外部サービス連携を勝手に追加しない
- アフィリエイトリンクをハードコードしない
- 収益を保証する表現や、怪しい副業感のある文言を追加しない
- `src/data/questions.ts`、`src/data/results.ts`、`src/data/cta.ts` の責務を混ぜない
